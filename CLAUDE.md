# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

SiftVault is an RSS aggregator (MVP), a TypeScript monorepo managed with npm workspaces (`client`, `server`). Requires Node >=24 (see `.nvmrc`).

- **client** — TanStack Start (file-based routing) + TanStack Query (server state) + Zustand (client state), Tailwind CSS, Shadcn/ui
- **server** — Fastify + Mongoose, MongoDB

MVP scope: user auth (register/login), RSS feed subscriptions (add/remove), parsing feed titles & links ("Title — Source"), read-status tracking, responsive UI.

## Commands

All commands are run from the repo root via the `Makefile` (run `make help` to list them). Under the hood these call `npm run <script> --workspaces --if-present` or target a single workspace with `-w @siftvault/client` / `-w @siftvault/server`.

```bash
make install       # npm install (installs both workspaces)
make dev-client     # vite dev server, http://localhost:3000
make dev-server     # tsx watch, http://localhost:3001
make dev            # both concurrently
make build           # tsc/vite build, both workspaces
make test            # vitest run, both workspaces
make lint            # eslint, both workspaces
```

Single test file (client only has tests today):

```bash
npm run test -w @siftvault/client -- src/routes/index.test.tsx
# or, from client/
npx vitest run src/routes/index.test.tsx
```

Single workspace lint/typecheck:

```bash
npm run lint -w @siftvault/client        # or @siftvault/server
npm run typecheck -w @siftvault/client   # or @siftvault/server
```

A Husky `pre-commit` hook runs `npm run lint` then `npm test` — commits are blocked on lint or test failure.

## Architecture

### Client (TanStack Start)

- Routes live in `client/src/routes/`, file-based. `client/src/routeTree.gen.ts` is **auto-generated by TanStack Router — never edit it**; it's regenerated on dev/build and is gitignored.
- `client/src/routes/__root.tsx` defines the root route (`createRootRoute`) — HTML shell, `<head>` metadata, and wraps `<Outlet />`.
- Route modules export a named `Route` via `createFileRoute(...)`, with `loader` for data fetching and `component` for the view. Route components are conventionally defined *after* the `Route` export in the same file (safe due to function hoisting) — the ESLint config explicitly allows this (`@typescript-eslint/no-use-before-define` has `functions: false`).
- `client/src/config.ts` exports `API_BASE_URL`, read from `VITE_API_URL` with a localhost fallback. Server calls from route loaders go through this.
- `client/src/router.tsx` wires `routeTree` into `createRouter`.
- Tests use Vitest + jsdom + Testing Library (`vitest.config.ts`, `vitest.setup.ts`), co-located as `*.test.tsx` next to the source file.

### State management (client)

- **TanStack Query** for server/async data (fetching, caching, invalidation).
- **Zustand** for all client state. Never use `useState` — always a Zustand store, split by domain (e.g. auth store, feeds store, UI store), not one global blob. Enforced by ESLint (`no-restricted-imports` / `no-restricted-properties` in `client/.eslintrc.cjs`) — importing or calling `useState` is a lint error, not just a convention.
- Select only the slice a component needs, and use shallow comparison when selecting multiple values, to avoid re-renders:
  ```ts
  const name = useStore((s) => s.name);
  const { a, b } = useStore((s) => ({ a: s.a, b: s.b }), shallow);
  ```
- Get actions without subscribing — pull them via `getState()` or a stable actions selector, not a value selector that re-renders the component.
- Read/write state outside React when reactivity isn't needed (event handlers, effects) via `getState()`/`setState()` directly, e.g.:
  ```ts
  const handleClick = () => {
    const currentName = useStore.getState().name;
    useStore.setState({ name: 'new name' });
  };
  ```
- Memoize derived/computed values (selector-level or `useMemo`), don't recompute inline on every render.
- Prefer declarative style and immutable updates (spread/produce, no in-place mutation) in store actions.

### Code style

- Import order: external (`node_modules`) imports first, then a blank line, then internal/relative imports. E.g.:
  ```ts
  import { useEffect } from 'react';
  import { create } from 'zustand';

  import { API_BASE_URL } from '../config';
  import { useAuthStore } from './auth-store';
  ```
  Not currently enforced by ESLint (Airbnb's `import/order` groups builtin/external/internal together with no blank line required) — this is a manual convention until/unless it gets wired into `import/order`'s `pathGroups`/`newlines-between`.

- No magic strings for user-facing copy (button/label text, headings, descriptions, placeholders, error copy). The client uses `i18next` + `react-i18next` for this (`client/src/i18n/i18n.ts`, initialized once via a side-effect import in `client/src/router.tsx` so it's ready on both the SSR and client render paths). All copy lives in `client/src/i18n/locales/en.json`, keyed by feature/component (e.g. `siteHeader.signIn`, `login.description`). Components call `useTranslation()` and read strings via `t('siteHeader.signIn')` instead of inlining `'Sign in'`. Interpolated copy uses `{{placeholder}}` syntax in the JSON and a second argument to `t`, e.g. `t('login.description', { appName: t('app.name') })`.
  - Only one locale (`en`) exists today — this is about eliminating magic strings and leaving a real path to add locales later, not shipping a language switcher.
  - Tests should assert against the same translation strings (import `en.json` directly, or match on the rendered text) rather than re-typing copy, so copy and tests can't drift apart.
  - This does **not** apply to `client/src/components/ui/*` — those are shadcn-vendored primitives regenerated by the CLI (`npx shadcn add ...`); leave their inline copy (e.g. dialog's "Close") as-is.

### Server (Fastify)

- Single entrypoint `server/src/index.ts` — creates the Fastify app, registers `@fastify/cors` scoped to the client origin, defines routes, and starts listening on `PORT` (default 3001).
- Mongoose/MongoDB is part of the intended stack (per README) but not yet wired up in `src/index.ts`.

### Linting

ESLint 8.x pinned deliberately — `eslint-config-airbnb`/`eslint-config-airbnb-typescript` don't yet support ESLint 9/10's flat config. Each workspace has its own `.eslintrc.cjs`:

- `client`: `airbnb` + `airbnb-typescript` + `airbnb/hooks`, type-aware via `client/tsconfig.json`.
- `server`: `airbnb-base` + `airbnb-typescript/base`, type-aware via `server/tsconfig.json`.

Both enforce semicolons. Notable overrides from stock Airbnb (in each `.eslintrc.cjs`): `import/prefer-default-export` off (TanStack route files and shared modules use named exports by convention), `import/extensions` off, and client-only `react/react-in-jsx-scope`/`react/require-default-props` off (new JSX transform + TypeScript types make these unnecessary).
